# Issue Report: Frontend Redesign Branch QA

**Branch:** `claude/frontend-redesign-plan-iXciv`

## Summary

Tested the dark-theme frontend redesign by creating a "protein_engineering" community, starting a deliberation thread, and navigating all major pages. The dark theme and overall layout are solid, but there are several functional issues ranging from startup UX to crash bugs.

---

## Issue 1: Platform requires manual initialization before any use

**Severity:** High (UX / onboarding blocker)
**Location:** Backend startup sequence

### Description

After a fresh `docker compose up`, the app loads but every API call returns `503 Service Unavailable` with `"Platform not initialized. Call POST /api/platform/init first."` The user must either click "Initialize Platform" in Settings or manually call `POST /api/platform/init` before the app is functional. Creating a community, viewing agents, or any data-dependent operation fails silently or with toast errors until this is done.

### Root cause

The `PlatformManager` is created at startup but intentionally **not initialized** (lazy init pattern). The lifespan hook in `src/colloquip/api/__init__.py:34-47` only sets up the database -- it does not call `pm.initialize()`. The comment on line 78 explicitly says:

```python
# Platform manager (lazy init -- call POST /api/platform/init to activate)
```

The `initialize()` method loads agent personas from YAML files into memory via `self.registry.load_from_personas()`. All platform route handlers check `pm._initialized` and return 503 if false (`src/colloquip/api/platform_routes.py:119-123`).

### Impact

- First-time users see an empty app with no indication of what to do
- The "Create Community" button is visible but fails with a cryptic 503 toast
- The home page shows "No communities yet" even though the real problem is uninitialized platform
- The only hint is a small toast notification that auto-dismisses

### Recommendation

Either:
1. **Auto-initialize on first request** -- add a middleware or dependency that calls `initialize()` on the first API hit if `_initialized` is false
2. **Initialize during startup** -- call `pm.initialize()` in the lifespan hook after database setup
3. **Block the UI** -- if manual init is intentional, show a prominent full-screen onboarding modal on first load instead of letting users interact with a broken app

## Issue 2: Community creation only assigns the Red Team agent

**Severity:** High (functional bug)
**Location:** Frontend `CreateCommunityDialog.tsx` + Backend `registry.py`

### Description

When creating the "protein_engineering" community with thinking type "ideation" and primary domain "protein engineering", only 1 agent was assigned: **Adversarial Analyst (Biology)** (red team). The **Protein Engineering Lead** agent -- which has `agent_type: protein_engineering`, expertise in `directed_evolution`, `rational_design`, etc. -- was not included despite being the most relevant agent in the pool.

### Root cause

The `CreateCommunityDialog` (`web/src/components/dialogs/CreateCommunityDialog.tsx:77-83`) sends the following fields to the API:

- `name`, `display_name`, `description`, `thinking_type`, `primary_domain`

It does **not** send `required_expertise` or `optional_expertise`.

The backend recruitment logic in `registry.py:264-289` (`recruit_for_subreddit()`) iterates over `required_expertise` to find matching agents. When the list is empty (as it always is from the frontend), **no agents are matched**. The only agent that gets added is the mandatory red team agent (`registry.py:307-316`), which is always included regardless of expertise matching.

The red team selection logic in `_recruit_red_team()` (`registry.py:320-351`) first tries a domain-specific match (checking if the domain substring appears in the agent type), then falls back to any available red team agent. "protein_engineering" doesn't match "red_team_biology", so it falls back to picking the first available red team agent.

### Impact

- Communities are created with only a red team agent and no domain experts
- Deliberations would have only adversarial analysis with no substantive expertise
- The platform's core value (multi-agent deliberation with diverse experts) is broken

### Recommendation

1. **Add expertise fields to the Create Community dialog** -- let users select required/optional expertise areas, or at minimum auto-populate based on `primary_domain`
2. **Infer expertise from domain** -- the backend could auto-map `primary_domain: "protein engineering"` to `required_expertise: ["protein_engineering"]` when no explicit expertise is provided
3. **Show agent preview** -- before creating, show which agents will be recruited so users can verify

## Issue 3: Agent detail page crashes

**Severity:** High (crash)
**Location:** `web/src/routes/agents/$agentId.tsx`

### Description

Navigating to any agent's detail page (e.g., `/agents/1e1c05c4-d8d1-49b3-afb5-aed42cc91442` for Protein Engineering Lead) crashes the entire app with:

```
Something went wrong!
Cannot convert undefined or null to object
```

The page renders a full-screen error boundary with no way to recover except navigating away.

### Root cause

The component fetches `/api/agents/{id}/calibration` which returns 503 (no calibration data exists yet). The component then attempts to call `Object.keys()` or `Object.entries()` on the undefined/null calibration response without a null check, causing the crash.

### Impact

- No agent detail pages are viewable
- Clicking any agent card from the Agent Pool grid crashes the app
- Users cannot inspect individual agent configurations, expertise, or history

### Recommendation

Add null/undefined guards before iterating over calibration data. The component should render a graceful empty state ("No calibration data yet") when the endpoint returns an error or empty response.

## Issue 4: Deliberation thread view fails to load session history

**Severity:** Medium (functional)
**Location:** Thread/deliberation view component

### Description

After successfully creating a deliberation thread ("De novo design of thermostable lipase variants"), the app navigates to the thread view at `/c/protein_engineering/thread/{id}`. The page layout renders correctly (phase progress sidebar, agents panel, cost summary), but the main content area shows:

```
Error Loading Thread
Error: Failed to load session history
```

The API call to `/api/deliberations/{id}/history` returns an error.

### Impact

- Thread content is not viewable after creation
- The deliberation UI layout is visible but non-functional
- Phase progress, agent activity, and post feed are all empty

### Recommendation

Investigate whether the history endpoint requires the deliberation to actually run before returning data. If so, the UI should show a "Deliberation starting..." or "Waiting for first post..." state instead of an error.

## Issue 5: Watchers endpoint returns 503

**Severity:** Low (expected for now)
**Location:** `/api/subreddits/{name}/watchers`

### Description

The Watchers tab on the community page calls `/api/subreddits/protein_engineering/watchers` which returns 503. The UI handles this gracefully, showing "No watchers configured" with a helpful description.

### Impact

Minimal -- the empty state is handled well. But the 503 response generates console errors and unnecessary toast notifications.

### Recommendation

Return 200 with an empty list instead of 503 when no watchers are configured. Reserve 503 for actual service unavailability.

## Issue 6: Minor pluralization bug on community cards

**Severity:** Low (cosmetic)
**Location:** Home page community card, community header

### Description

The community card on the home page displays "1 agents" and "1 threads" instead of "1 agent" and "1 thread".

### Impact

Cosmetic only, but makes the product feel unpolished.

### Recommendation

Add simple pluralization logic: `${count} ${count === 1 ? 'agent' : 'agents'}`.

## Overall Assessment

The dark theme redesign is a significant visual improvement. The layout, navigation, and component design are clean and consistent. However, the app has a critical onboarding gap (manual platform init), a core feature gap (community agent recruitment), and a crash bug (agent detail page) that need to be resolved before this branch is demo-ready.
